# Releasing Peek

One sandboxed build, two distribution channels:

| Channel              | Signing identity                     | Artifact                              |
| -------------------- | ------------------------------------ | ------------------------------------- |
| GitHub Releases      | Developer ID Application + Installer | `.dmg` + `.pkg`, notarized + stapled  |
| Mac App Store        | Apple Distribution                   | Archive uploaded via Xcode Organizer  |

Same `peek` target, same `Release` configuration, same `com.oldsalt.peek` bundle ID. Only the signing identity and distribution mechanism differ. Both builds have App Sandbox on, hardened runtime on, and the same entitlement set (`com.apple.security.app-sandbox` + `com.apple.security.network.server`).

---

## One-time setup

Re-run after macOS reinstalls or DerivedData wipes:

1. **Developer ID certs in Keychain** (for the GitHub channel). `security find-identity -v | grep "Developer ID"` should show both `Developer ID Application: …` and `Developer ID Installer: …`. Add via Xcode → Settings → Accounts → Manage Certificates.
2. **Apple Distribution cert in Keychain** (for the App Store channel). Same Xcode Manage Certificates dialog.
3. **Notarization profile** (for the GitHub channel). `xcrun notarytool store-credentials peek-notary --apple-id YOUR_APPLE_ID --team-id 346JJCHZP7` — prompts for an app-specific password (generate at appleid.apple.com → Sign-In & Security → App-Specific Passwords).
4. **App Store Connect record** for bundle ID `com.oldsalt.peek`. Set the subtitle without Apple trademarks (no "Mac" / "macOS" / "AirDrop" / etc. — those trigger 5.2.5 IP rejections).

### App Store Connect strings

Lock these in at submission time and don't drift between releases without good reason:

| Field | Value | Notes |
|---|---|---|
| Subtitle (≤30 chars) | **`Local windows for AI agents.`** | 28 chars. No trademarked words. |
| Category | Utilities | matches `LSApplicationCategoryType` |
| Privacy policy URL | `https://peek.dort.zone/privacy` | content lives at `PRIVACY.md` in this repo |
| Support URL | `https://github.com/just-an-oldsalt/peek/issues` | |
| Marketing URL (optional) | `https://peek.dort.zone/` | |

Reviewer note (paste into the Resolution Center on first submission):

> Peek runs a localhost-only MCP endpoint (bound to 127.0.0.1) so AI agents already configured on the user's Mac can request window captures via the Model Context Protocol. The endpoint requires a bearer token (generated in-app, stored in Keychain) and a per-app user approval prompt before any capture. No network traffic leaves the device. The `com.apple.security.network.server` entitlement is required only to bind the loopback listener; Peek does not have any client networking entitlements.

## GitHub release workflow

1. **Bump version.** Edit `MARKETING_VERSION` and `CURRENT_PROJECT_VERSION` in `peek.xcodeproj/project.pbxproj` (each appears multiple times — search-and-replace both numbers). Commit:
   ```
   git commit -am "Bump version to X.Y (build N)"
   ```

2. **Build, sign, notarize, package.** From the repo root:
   ```
   ./scripts/release.sh X.Y
   ```
   Takes 3–15 minutes — two notarization rounds (`.dmg`, `.pkg`). The script verifies the version arg matches `MARKETING_VERSION`, so it'll bail loudly if you forgot step 1.

3. **Tag and publish to GitHub.** The script prints the exact commands; copy-paste:
   ```
   git tag -a vX.Y -m "Peek X.Y"
   git push origin vX.Y
   gh release create vX.Y \
       --title "Peek X.Y" \
       --generate-notes \
       build/release/artifacts/peek-X.Y.dmg \
       build/release/artifacts/peek-X.Y.pkg \
       build/release/artifacts/SHA256SUMS
   ```

## Mac App Store workflow

Same version bump (step 1 above). Then:

1. **Archive in Xcode.** Product → Archive (default scheme + `Release` configuration).
2. **Distribute App** from the Organizer window → App Store Connect → Upload. Xcode validates the signature, sandbox entitlements, `PrivacyInfo.xcprivacy`, and uploads to App Store Connect.
3. In App Store Connect, attach the new build to a version, fill in release notes, submit for review.

### Common review snags

- **5.2.5 (Apple trademarks)** — anything in the subtitle/keywords/screenshots that uses "Mac", "macOS", "App Store", "AirDrop", etc. as a product modifier. Use generic terms.
- **`network.server` entitlement scrutiny** — the reviewer may ask "why does a screenshot utility need a server?" Reply in the Resolution Center: "Peek runs a localhost-only MCP endpoint (bound to 127.0.0.1) so AI agents connected to the user's machine can request window captures via the Model Context Protocol. All requests require a bearer token generated by the user in Settings, and the user must approve each new app the agent wants to capture. No network traffic leaves the device."
- **Screen Recording usage** — the privacy nutrition label in App Store Connect should describe Screen Recording as user-initiated, on-device, and not collected. The `PrivacyInfo.xcprivacy` ships zero data-collection types, which matches.

## Failure modes

| Symptom | Likely cause |
|---|---|
| Notarization rejected on `.app` | `xcrun notarytool log <SUBMISSION_ID> --keychain-profile peek-notary` for the actual reason — usually a missing Hardened Runtime flag or a bad entitlement. |
| MAS archive upload fails on missing entitlements | Confirm `peek/peek.entitlements` is the file referenced by `CODE_SIGN_ENTITLEMENTS` in Build Settings; it must contain `com.apple.security.app-sandbox=true`. |
| Gatekeeper rejects the stapled artifact | The `.app` was modified after stapling. Re-archive from scratch. |
| MCP server can't bind on launch | Ports 11474–11479 all taken by another local service. The server logs the chosen port via `log show --predicate 'subsystem=="com.oldsalt.peek" AND category=="mcp"'`. |
| App Store reviewer cannot reach the MCP endpoint | The reviewer's sandboxed test rig doesn't run a local AI client; ship a screen recording in the review notes demonstrating the click-to-clipboard menu (it requires no MCP). |
